# Secret Rotation Runbook

Use this runbook when Gitleaks flags local environment files or when a credential might have been exposed.

Do not paste secret values into chat, issues, commits, or logs. Rotate in the provider dashboard, update local env files, redeploy, then revoke the old credential.

## Current Local Findings

The repository history scan is clean with the existing `.gitleaksignore` baseline. Filesystem scans can still find secrets in ignored local files because `--no-git` ignores `.gitignore`.

Local files that were flagged:

- `.env.local`
- `apps/agent/.env.local`
- `apps/agent/.env.production.local`

Detected variable names:

- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `TEST_SUPABASE_SERVICE_ROLE_KEY`
- `BASESCAN_API_KEY`
- `DEPLOYER_PRIVATE_KEY`
- `GITHUB_MCP_TOKEN`
- `OPENROUTER_API_KEY`
- `VERCEL_OIDC_TOKEN`

## Scan Commands

Use the main config for repo/history scans:

```sh
gitleaks detect --source . --redact=100 --no-banner
```

Use the local config only when you intentionally scan the working directory as plain files:

```sh
gitleaks detect --source . --no-git --config .gitleaks.local.toml --redact=100 --no-banner
```

Do not use `.gitleaks.local.toml` in CI. It suppresses ignored env files for local filesystem scans only.

## Provider Rotation Checklist

### Supabase

Rotate any exposed Supabase service-role, anon, or test service-role keys in the Supabase dashboard.

1. Open the Supabase project dashboard.
2. Rotate or create replacement API/JWT keys according to the project key type.
3. Update local env files and deployment env vars.
4. Redeploy services that use the keys.
5. Revoke the old key after the new deployment is working.

Never expose `SUPABASE_SERVICE_ROLE_KEY` or any secret key to browser code. `NEXT_PUBLIC_` variables are bundled into the frontend.

### OpenRouter

1. Open OpenRouter API key settings.
2. Create a replacement key for `OPENROUTER_API_KEY`.
3. Update `apps/agent/.env.local` and any deployment env var.
4. Revoke/delete the old OpenRouter key.

### GitHub

1. Open GitHub personal access token settings.
2. Create a new fine-grained token with only the required scopes.
3. Update `GITHUB_MCP_TOKEN` locally.
4. Delete the old token.

### BaseScan

1. Open the BaseScan API key dashboard.
2. Create a new API key for `BASESCAN_API_KEY`.
3. Update local and deployment env vars.
4. Remove the old API key.

### Vercel

`VERCEL_OIDC_TOKEN` is generated by Vercel and is short-lived. Do not copy it into persistent local env files unless it is required for a short local test.

For static Vercel env vars:

1. Open the Vercel project settings.
2. Replace the affected environment variable values.
3. Redeploy the affected environments.
4. Revoke the old credential in the original provider dashboard.

### Deployer Wallet

`DEPLOYER_PRIVATE_KEY` is a blockchain wallet private key, not a provider dashboard secret.

1. Create a new wallet.
2. Fund it with enough gas for the target chain.
3. Grant required on-chain roles to the new wallet where supported.
4. Re-register or transfer agent ownership where required by the contract model.
5. Update local and deployment env vars.
6. Stop using the old private key.

For the current `VaeltusRegistry`, `resetCircuit` requires the agent owner. If the old private key owns an agent and there is no transfer-owner function, create/register a replacement agent under the new wallet or add an explicit ownership-transfer flow before relying on the new wallet.
